Authentication

Which authentication methods can be used with the Coinbase API?

We offer two different methods for authentication API requests. These methods are:

  1. API Key + Secret - Best for accessing your own account
  2. OAuth - Best for accessing other users' accounts

We previously offered a simple API Key authentication method as well, that has been deprecated and will be discontinued in August 2014.

How can I use these authentication methods?

More details on how to authenticate requests is available on the página de autentificación.

You can also read our guía detallada sobre la integración OAuth2

How do permissions work with API credentials?

When generating API credentials for an account, you will be given the option to select specific permissions. The credentials generated will only have access to the permissions selected here. More information about permissions and how they work is available at https://www.coinbase.com/docs/api/permissions

How can I send POST requests with the HMAC/Secret authentication method?

To send a POST request with the HMAC/Secret authentication scheme, you will need to append the POST request's body to the end of the signed string. Example:

If you are signing the following request:
  Nonce: 1234
  API Call URL: https://api.coinbase.com/v1/accounts
  POST Body: { "account": { "name": "Savings Wallet" } }

You will need to sign the following data:
  1234https://api.coinbase.com/v1/accounts{"account":{"name":"Savings Wallet"}}

Refunds

How can I refund an order with the API?

To refund an order with the API, use the /reembolsar llamada . This call requires two parameters:

  1. The order code, which you'll need to include in the request URL: https://api.coinbase.com/v1/orders/ORDERCODE/refund
  2. The refund currency, which can be either BTC or USD, included in the request body. If BTC is selected, we will refund the original BTC payment amount. If USD is selected, we will refund an amount equivalent to the USD-denominated order amount.

The documentation for this API call, as well as more information about the required parameters, can be found in our full API reference aquí

How is my website notified when an order is completed?

Coinbase offers a callback notification system. If enabled, our system will notify your website when a customer completes, cancels, or mispays their order. More information about these notifications can be found at the following documentation page: https://www.coinbase.com/docs/merchant_tools/callbacks

When your daily cashout is processed (usually around 2PM PST), our system will send your website a callback notification with the transaction details.

Mispayments

I saw a mispayment on the orders page of my account (https://www.coinbase.com/orders). What does this mean?

A mispayment is an order payment that was received under abnormal circumstances. Two triggers will cause a mispayment to occur:

  1. The customer's payment was received after the 10-minute order timeout expired.
  2. The customer's payment was greater than or less than the actual order amount.

Mispayments can usually be handled by either refunding or fulfilling the customer's order.

In the case of (1), where a payment was received after the timeout, many merchants choose to fulfil the order. In the case of (2) - a mispayment triggered by an under/over-payment - it is often easiest to refund the customer's order and have them create a new one.

What are the different possible values for "status" on mispayments?

Mispayments can have several different status indicators.

  • Converted to USD: You have auto-cashout enabled for orders and mispayments. These BTC are waiting to be bundled into a daily bank transfer.
  • Cashed Out: You have auto-cashout enabled for orders and mispayments. The BTC have been bundled into a daily bank transfer.
  • Redirected: You have auto-cashout enabled for orders but not mispayments. The BTC originally went directly to Coinbase's internal account, and were then sent to your Coinbase account.
  • Received: You have auto-cashout enabled neither for orders nor mispayments. The BTC have gone straight to your Coinbase account
  • Pending: Waiting for confirmations from the Bitcoin network (blockchain) before further action.

Troubleshooting

Why did I get a 401 Unauthorized response?

If you received a 401 Unauthorized response on one of your requests, it is likely that an issue is present in your authentication parameters. Make sure that:

  1. Your API key and secret is enabled (or your OAuth grant is not revoked)
  2. The parameters were included in the correct place (e.g. in the request headers)
  3. With the key and secret authentication scheme, make sure that the correct data was signed.

If all of the above factors are normal, shoot us over an email at api@coinbase.com and we'll be happy to take a look.

What are all the possible response/error types generated by the API?

Although many different responses are possible, all will conform to one of a few formats. These formats are all listed below, along with their HTTP status codes:

  • 404
    • A request is made to an API endpoint that does not exist. The body of this response is Coinbase's 404 page. One common cause of this issue is attempting to GET an endpoint that expects POST requests (and vice-versa).
  • 401
    • A request is made to an API that exists, but is unauthorized. These responses follow the format of {"error":"reason"}. A few common causes of this error are as follows:
    • Attempting to use an API Key (or OAuth2 grant) that does not have suitable permissions. As an example, calling the POST /api/v1/transactions/send_money endpoint with an API key that does not have the send permission.
    • Attempting to use a nonce that is less than the previous one used.
  • 200

    • The API is successfully communicated with. These responses follow the format of {"success":[true or false],["error":"reason" or returned data]}. A few common causes of this response are as follows:

    • The request is formed correctly, but with parameters referencing data that doesn't exist, or for which you don't have authorization.
      • {"success":false,"error":"Order not found with that id"}
    • The request is formed correctly, and the requested data is returned.
      • {"success":true,"address":"1FG4EcuqQ2zPwNrzRSdKJ8kvmbFhbQfYsu","callback_url":null,"label":null}

Why aren't redirect URLs working on my payment buttons/iframes?

Redirect URLs will only be used for payment pages hosted on Coinbase.com: https://www.coinbase.com/merchant_tools?link_type=hosted.

These URLs will not be applied to buttons or iframes, as doing so would require unnecessarily invasive Javascript. If you need to apply redirection URLs to these, you can set up your own Javascript to listen for the coinbase_payment_complete Javascript event, and redirect your users when that fires. You can see an example of this in the Using Your Own Button And Custom Javascript Events section of our documentation.

How do I receive callback data in PHP?

$data = file_get_contents("php://input");

Many PHP developers try to use the $_POST variable with Coinbase callbacks, only to find they're not getting any data. $_POST works only when the POSTer has specified a content type of application/x-www-form-urlencoded and multipart/form-data.

Coinbase sends everything as application/json. php://input returns the entire body of the POST request regardless of content type.

Why isn't my AJAX request to the API working?

Coinbase's Access-Control-Allow-Origin won't allow authenticated browser-based AJAX requests, even if you try to do so with JSONP (although unauthenticated requests, like the "prices" API, will work).

Browser extensions (such as Postman) are not necessarily subject to the same rules and may be able to do so.
